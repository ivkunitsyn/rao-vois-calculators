# Система калькуляторов РАО и ВОИС

Веб-система специализированных калькуляторов для расчета справочного вознаграждения по правилам РАО и ВОИС. Репозиторий восстановлен в общих чертах из внутреннего проекта `sonicscout/rnd/rnk`: перенесены основные backend-модули калькуляторов, интерфейс и документация по логике, без тяжелых рабочих таблиц и без чувствительных данных.

## Что входит

| Модуль | Назначение |
| --- | --- |
| `calculators/rao_tv` | Расчет по телевизионному использованию для РАО. |
| `calculators/rao_radio` | Расчет по радиовещанию для РАО. |
| `calculators/rao_events` | Расчет для мероприятий / публичного исполнения по РАО. |
| `calculators/vois_tv` | Расчет по телевизионному использованию для ВОИС. |
| `calculators/vois_radio` | Расчет по радиовещанию для ВОИС. |
| `calculators/vois_events` | Расчет для мероприятий / публичного исполнения по ВОИС. |
| `interface/` | Единый HTML-интерфейс калькуляторов. |
| `docs/` | Архитектурные заметки и описание логики отдельных модулей. |
| `app.py` | Точка входа веб-приложения. |

## Назначение

Система помогает пользователю рассчитать ориентировочную сумму вознаграждения по выбранной категории использования. Она собирает исходные параметры, валидирует их, применяет правила конкретного калькулятора и возвращает результат справочного характера.

Типовые сценарии:

- расчет для телеканала;
- расчет для радиостанции;
- расчет по публичному мероприятию;
- раздельная логика для РАО и ВОИС;
- подготовка результата в человекочитаемом виде;
- выгрузка/рендеринг результата для радиокалькулятора.

## Архитектура

Проект устроен модульно: каждый калькулятор живет в отдельном каталоге и имеет собственный backend.

```text
calculators/
  rao_tv/
    backend/
      api.py
      calculator.py
      rao.py
  rao_radio/
    backend/
    result/
  rao_events/
    backend/
  vois_tv/
    backend/
  vois_radio/
    backend/
  vois_events/
    backend/
interface/
docs/
app.py
```

Общая схема обработки:

1. Пользователь выбирает тип калькулятора.
2. Frontend собирает параметры формы.
3. Backend-модуль соответствующего калькулятора валидирует данные.
4. Calculation core применяет ставки, коэффициенты и правила.
5. API возвращает структурированный результат.
6. Интерфейс показывает итог и пояснения.

## Калькуляторы

### РАО ТВ

Самый крупный модуль в проекте. Включает API, расчетную логику и вспомогательные функции для обработки пользовательских данных, каналов, ставок и параметров расчета.

Ключевые файлы:

- `calculators/rao_tv/backend/api.py`;
- `calculators/rao_tv/backend/calculator.py`;
- `calculators/rao_tv/backend/rao.py`.

Рабочие Excel-таблицы ставок не включены в репозиторий. Для production их нужно хранить отдельно и подключать через конфигурацию/volume.

### РАО радио

Калькулятор радиовещания РАО. Помимо backend-расчета содержит слой рендеринга результата:

- HTML;
- plain text;
- DOCX.

Ключевые файлы:

- `calculators/rao_radio/backend/calculator.py`;
- `calculators/rao_radio/backend/rao_radio.py`;
- `calculators/rao_radio/result/`.

### РАО мероприятия

Модуль расчета для публичных мероприятий и событий.

Ключевой файл:

- `calculators/rao_events/backend/calculator.py`.

### ВОИС ТВ

Калькулятор телевизионного использования для ВОИС. Логика описана дополнительно в `docs/VOIS_TV_LOGIC.md`.

Ключевой файл:

- `calculators/vois_tv/backend/calculator.py`.

### ВОИС радио

Калькулятор радиовещания для ВОИС.

Ключевой файл:

- `calculators/vois_radio/backend/calculator.py`.

### ВОИС мероприятия

Калькулятор публичных мероприятий для ВОИС. Логика описана дополнительно в `docs/VOIS_EVENTS_LOGIC.md`.

Ключевой файл:

- `calculators/vois_events/backend/calculator.py`.

## Интерфейс

`interface/index.html` содержит единый web UI. В нем есть формы, переключение режимов и клиентская логика для отправки параметров в backend.

Важно: интерфейс перенесен как часть общего представления системы. Перед production-запуском его нужно проверить на соответствие актуальным API endpoints и убрать неиспользуемые административные блоки, если они не нужны в публичной версии.

## Данные и ставки

Внутренний проект содержал Excel-таблицы и рабочие data-файлы. В этот репозиторий они не перенесены намеренно:

- могут содержать рабочие данные;
- утяжеляют репозиторий;
- должны версионироваться отдельно от кода;
- для production лучше подключать их через protected storage или deployment volume.

Ожидаемые внешние данные:

- таблицы ставок РАО;
- таблицы ставок ВОИС;
- справочники организаций/ИНН, если используются;
- upload logs и пользовательские расчеты.

## Конфигурация

Рекомендуемый подход:

```env
RKN_RAO_TV_DATA_DIR=/path/to/rao_tv/data
RKN_VOIS_TV_DATA_DIR=/path/to/vois_tv/data
YANDEX_API_KEY=
YANDEX_MODEL_URI=
ADMIN_USERS_JSON=
```

Все пароли, API keys и рабочие таблицы должны жить вне Git.

## Запуск

```bash
python3 -m venv .venv
source .venv/bin/activate
pip install -r requirements.txt
python app.py
```

Если приложение использует FastAPI/Flask entrypoint внутри `app.py`, production-запуск лучше делать через `uvicorn`/`gunicorn` или systemd unit с явным `WorkingDirectory`.

## Безопасность

- Hardcoded admin-пароли из исходного проекта заменены на `<REDACTED>`.
- Excel/data-файлы не перенесены.
- `.env`, логи, базы и пользовательские выгрузки должны быть исключены из Git.
- Для production нужно настроить нормальный механизм пользователей и паролей через environment/secret storage.
